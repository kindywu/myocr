# 药品录入功能 — 实施方案

> 基于 `prototype/` 交互原型，输出可落地的技术实现方案。
>
> **核心原则：** OCR 帮填，手填兜底。拍照识别为主路径，手动录入为兜底路径。

---

## 1. 用户流程总览

```
                        ┌──────────────────┐
                        │     录入方式      │
                        │   （入口选择页）  │
                        └──────┬───────────┘
                               │
              ┌────────────────┼─────────────────┐
              │  「拍照识别」   │                  │  「手动填写」
              ▼                │                  ▼
    ┌───────────────────┐     │      ┌──────────────────────┐
    │     拍照采集       │     │      │      手动录入          │
    │  步骤 1/3 — 正面  │     │      │  （完整表单，自行填写） │
    └───────┬───────────┘     │      └──────────┬───────────┘
            │ 拍摄完成        │                  │ 保存
            ▼                 │                  ▼
    ┌───────────────────┐     │      ┌──────────────────────┐
    │    多角度补全      │◄────┘      │      信息确认          │
    │  进度视图 + 补拍   │             │  （OCR 结果 + 编辑）  │
    └───────┬───────────┘             └──────────┬───────────┘
            │  「查看结果」                       │  「保存药品信息」
            ▼                                    ▼
    ┌───────────────────┐              ┌──────────────────────┐
    │     信息确认       │              │      保存完成          │
    │  （OCR 结果+编辑）  │              │  （成功反馈页）       │
    └───────────────────┘              └──────────────────────┘
            │  「保存药品信息」
            ▼
    ┌───────────────────┐
    │     保存完成       │
    │  （成功反馈页）    │
    └───────────────────┘
```

**关键路径：**

| 路径 | 步骤 | 说明 |
|------|------|------|
| **主路径（OCR）** | 录入方式 → 拍照采集 → 多角度补全 → 信息确认 → 保存完成 | 用户拍照，OCR 自动识别，确认后保存 |
| **兜底路径（手动）** | 录入方式 → 手动录入 → 信息确认 → 保存完成 | 用户自行填写全部字段 |
| **混合路径** | 手动录入 → 去拍照补全 → 拍照采集 → ... | 手动填写中切换到拍照补全 |

## 2. 核心数据模型

### 2.1 药品信息（DrugInfo）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `drugName` | string | ✅ | 药品名称 |
| `expiryDate` | string | ❌ | 有效期，格式 `yyyy-MM` |
| `manufacturer` | string | ❌ | 生产厂家 |
| `batchNumber` | string | ❌ | 批号/注册号 |

### 2.2 字段识别状态（FieldStatus）

| 状态 | 含义 |
|------|------|
| `pending` | 尚未拍摄/识别 |
| `recognized` | OCR 已识别出值 |
| `manual` | 用户手动填写 |
| `confirmed` | 用户已确认 |

### 2.3 采集会话（CaptureSession）

```
{
  sessionId: string
  status: 'capturing' | 'completed'
  fields: Record<fieldKey, FieldStatus>
  photos: PhotoInfo[]
  drugInfo: Partial<DrugInfo>
  source: 'ocr' | 'manual' | 'mixed'
}
```

## 3. 功能模块划分

### 模块 A：入口选择页

- 展示两个入口：**拍照识别**（主推）和 **手动填写**（兜底）
- 展示品牌标语「OCR 帮填，手填兜底」
- 简单决策，无复杂状态

### 模块 B：拍照采集

- 全屏取景器，带对准指南框
- 调用系统相机 / 文件选择（取决于平台形态）
- 拍摄后将照片暂存，传递给 OCR 处理
- 当前进度指示：步骤 1/3（正面）

### 模块 C：多角度补全

- 展示药品各字段的识别进度（已识别 ✅ / 待拍摄 📷）
- 支持对特定未识别字段单独补拍
- 提供「继续拍摄」和「查看结果」两个动作
- 核心交互：增量采集，逐字段覆盖

### 模块 D：信息确认

- 展示所有 OCR 识别结果的表单（可编辑）
- 展示已拍摄的照片缩略图
- 用户可以修改任意字段的值
- 「保存药品信息」按钮提交最终数据

### 模块 E：手动录入

- 完整的表单页面，供用户自行填写
- 包含导航到拍照流程的入口（「去拍照补全」）
- 表单包含：药品名称、有效期、生产厂家、批号

### 模块 F：保存完成

- 成功反馈页，展示保存的数据摘要
- 「继续录入」→ 回到入口页
- 「完成」→ 返回上一级或关闭

## 4. 技术架构建议

### 4.1 分层架构

```
┌─────────────────────────────────────────────────┐
│                  展示层 (UI)                      │
│  录入方式 | 拍照采集 | 多角度补全 | 确认 | 手动   │
├─────────────────────────────────────────────────┤
│                  状态管理层                       │
│  采集会话状态 | 字段识别状态 | 编辑状态 | 提交    │
├─────────────────────────────────────────────────┤
│                  业务逻辑层                      │
│  OCR 调度 | 字段合并 | 数据校验 | 表单编排       │
├─────────────────────────────────────────────────┤
│                  服务层                          │
│  OCR 引擎 | 相机服务 | 存储服务 | 网络服务       │
├─────────────────────────────────────────────────┤
│                  基础设施层                      │
│  文件管理 | 权限管理 | 日志 | 配置              │
└─────────────────────────────────────────────────┘
```

### 4.2 推荐技术栈

| 层次 | 技术选型 | 考虑因素 |
|------|----------|----------|
| **UI 框架** | React / Vue / 原生小程序 | 取决于宿主平台；移动端优先 |
| **样式方案** | Tailwind CSS / UnoCSS | 原型使用了 Tailwind，匹配度高 |
| **状态管理** | Pinia (Vue) / Zustand (React) | 轻量、适合表单状态 |
| **OCR 引擎** | 百度 OCR / Tesseract / 自研 | 药品包装识别需要专用训练 |
| **相机接入** | 系统 Camera API / WebRTC | 原生能力优先 |
| **存储** | LocalStorage / IndexedDB / SQLite | 离线可用性 |
| **构建工具** | Vite / Webpack | 视平台而定 |
| **图标库** | Lucide | 与原型一致 |

### 4.3 可选架构方案

| 方案 | 说明 | 适用场景 |
|------|------|----------|
| **A. SPA 应用** | 单页应用，前端路由管理页面流转 | Web 或混合 App |
| **B. 多页应用** | 每个步骤独立页面，服务端渲染 | 低端设备、快速上线 |
| **C. 小程序版本** | 微信/支付宝小程序原生 | 中国移动端用户 |
| **D. 原生模块嵌入** | 作为模块嵌入已有 App | 已有 App 增加功能 |

**推荐：** 如果是新项目选 **A（SPA）**；如果是嵌入已有 App 选 **D**。

## 5. 状态管理与数据流

### 5.1 状态设计

```
采集会话 (CaptureSession)
├── currentStep: 'entry' | 'capture' | 'completion' | 'confirm' | 'manual' | 'saved'
├── captureRound: number          // 当前是第几轮拍摄（1/3, 2/3, 3/3）
├── photos: Photo[]               // 已拍摄照片列表
├── fieldResults: Map<FieldKey, { value, source, confidence }>
│   ├── drugName      → { value: '阿莫西林胶囊', source: 'ocr', confidence: 0.95 }
│   ├── expiryDate    → { value: '2026-09', source: 'ocr', confidence: 0.88 }
│   ├── manufacturer  → { value: null, source: 'pending', confidence: 0 }
│   └── batchNumber   → { value: 'H20093069', source: 'manual', confidence: 1.0 }
└── origin: 'ocr' | 'manual'
```

### 5.2 数据流方向

```
用户拍照 → 照片暂存 → OCR 识别 → 字段合并
                                    ↓
用户编辑 → 覆盖/补充字段 ← 多角度补全 (增量识别)
              ↓
        确认提交 → 持久化存储 → 成功反馈
```

### 5.3 关键状态转换

| 触发动作 | 从 | 到 | 副作用 |
|----------|----|----|--------|
| 选择拍照识别 | entry | capture | 初始化会话，打开相机 |
| 选择手动填写 | entry | manual | 初始化会话，展示空表单 |
| 拍照完成 | capture | completion | 提交 OCR 识别任务 |
| OCR 识别完成 | completion | completion(更新) | 更新字段识别状态 |
| 点击继续拍摄 | completion | capture(下一轮) | captureRound++ |
| 点击查看结果 | completion | confirm | 组装最终字段数据 |
| 点击去拍照补全 | manual | capture | 保留已填字段，打开相机 |
| 保存药品信息 | confirm | saved | 提交数据持久化 |
| 继续录入 | saved | entry | 重置会话 |
| 完成 | saved | (关闭/返回) | - |

## 6. 开发阶段规划

### Phase 1：核心骨架（预计 3-5 天）

- 搭建项目脚手架（路由、状态管理、UI 基础组件）
- 实现页面路由和导航（6 个页面的跳转逻辑）
- 实现采集会话状态管理（session store）
- 构建静态页面骨架（参照 prototype 的 HTML 结构）

### Phase 2：拍照与 OCR 链路（预计 5-7 天）

- 集成相机调用能力（拍照 / 图片选择）
- 封装 OCR 引擎调用（统一接口，便于切换供应商）
- 实现拍照 → OCR → 字段合并的数据管道
- 实现多角度补全的增量识别逻辑

### Phase 3：表单与确认（预计 3-4 天）

- 信息确认页面的表单渲染与编辑
- 手动录入页面的完整表单
- 数据校验逻辑（必填项、格式校验）
- 保存与持久化

### Phase 4：体验打磨（预计 3-5 天）

- 移动端适配与响应式优化
- 加载状态、空状态、错误状态处理
  - **加载中：** OCR 识别进度条 / 骨架屏
  - **空状态：** 未拍照时的引导提示
  - **错误状态：** OCR 识别失败 → 提示手动填写；拍照权限拒绝 → 引导开启
  - **边界情况：** 网络中断 → 本地暂存后同步；连续拍照 → 防止重复提交
- 动画过渡（参考原型的绿色勾选成功动画）
- 深色模式适配（原型已定义完整暗色色板）

### Phase 5：测试与上线（预计 3-4 天）

- 功能测试：全路径遍历（主路径 + 兜底 + 混合路径）
- 设备兼容性测试（不同屏幕尺寸、相机性能）
- OCR 准确率评估与调优
- 性能优化（图片压缩、缓存策略）

## 7. API 接口设计（建议）

### 7.1 OCR 识别

```
POST /api/ocr/recognize
  Request:  { image: base64 | file, sessionId: string }
  Response: { fields: Array<{ key, value, confidence }> }

POST /api/ocr/batch-recognize
  Request:  { images: Array<base64|file>, sessionId: string }
  Response: { fields: Record<string, { value, confidence, sourceImage }> }
```

### 7.2 药品数据

```
POST /api/drugs/save
  Request:  { sessionId, drugInfo, photos[], source }
  Response: { id, savedAt }

GET /api/drugs/:id
  Response: { id, drugInfo, photos[], createdAt }
```

### 7.3 离线策略

- 拍照 → 存入 IndexedDB 队列
- OCR 请求 → 网络可用时自动提交
- 保存 → 先写本地，后同步服务端
- 冲突 → 以本地最终确认版本为准

## 9. 关键设计决策记录

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| 页面流转方式 | 路由切换 / 单页内步骤切换 | 路由切换 | 各页面状态独立，便于深度链接和返回 |
| OCR 调用时机 | 拍照后立即 / 全部拍完统一 | 拍照后立即 | 增量反馈，用户可实时看到识别结果 |
| 照片存储 | 仅本地 / 上传服务端 | 本地 + 按需上传 | 离线可用，减少不必要的网络请求 |
| 字段覆盖策略 | 后识别的覆盖先识别的 / 人工确认后才覆盖 | 人工确认后才覆盖 | 避免高置信度早期结果被低置信度后期结果覆盖 |
| 表单校验时机 | 实时 / 提交时 | 实时 + 提交时双重 | 实时提示可快速修正，提交时兜底防止遗漏 |
| 多轮拍摄上限 | 3 轮 / 5 轮 / 不限 | 不限（引导 3 轮） | 用户应能自由补拍，但 UI 默认引导 3 轮完整覆盖 |

## 10. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| OCR 识别准确率不足 | 用户需要频繁手动纠正，体验下降 | ① 提供高置信度阈值过滤 ② 清晰的编辑入口 ③ 支持自定义 OCR 模型训练 |
| 拍照质量差（模糊/反光/遮挡） | OCR 无法识别 | ① 取景器对准引导 ② 实时质量检测 ③ 提示重新拍摄 |
| 网络离线 | OCR 服务不可用 | ① 本地缓存拍照图片 ② 队列化 OCR 请求 ③ 支持纯手动录入离线兜底 |
| 权限拒绝（相机） | 无法拍照 | ① 友好的权限引导弹窗 ② 直接降级到手动录入 ③ 支持从相册选择 |
| 药品包装信息布局不标准 | OCR 字段提取混乱 | ① 字段级置信度展示 ② 人工确认环节兜底 ③ 支持自定义裁剪区域 |
| 用户误操作（如连拍） | 重复识别，状态混乱 | ① 防抖处理 ② 照片去重 ③ 撤销能力 |

- **P1（重要）：** 多角度补全的增量识别逻辑正确
- **P1（重要）：** 表单编辑与数据合并无丢失
- **P1（重要）：** 全部 6 个页面的加载/空/错误状态覆盖
